# AR-перегляд килима

Прототип AR-фічі для магазину: користувач наводить телефон на підлогу і бачить килим
з реальною текстурою в натуральному розмірі, ставить його на місце й розглядає під будь-яким кутом.

## Підхід (і чому саме він)

Використовується **[`<model-viewer>`](https://modelviewer.dev/)** від Google — веб-компонент,
що делегує AR нативним рушіям ОС:

- **Android** → ARCore (Google Scene Viewer)
- **iOS** → ARKit (Apple Quick Look)

Детекцію підлоги, фіксацію об'єкта та жести (рух/поворот/масштаб пальцем) робить **сама ОС** —
це стабільно і працює однаково на обох платформах.

### Чому не «чистий» WebXR + Three.js

- **WebXR `immersive-ar` НЕ підтримується на iOS Safari** (станом на 2026). Тобто саморобний
  WebXR-AR = тільки Android. Для інтернет-магазину це втрата ~40% мобільного трафіку.
- Саморобний hit-test / reticle нестабільний; нативний ARCore/ARKit на порядок надійніше.
- Кастомні екранні кнопки в WebXR (через 3D-спрайти або ручний raycast) — джерело багів.
  У model-viewer керування нативне, кнопки не потрібні.

Єдина «ціна» model-viewer: потрібна **GLB-модель** (а не текстура на льоту). Тому килим
пакується у `.glb` тонким прямокутником із накладеною текстурою. На iOS model-viewer
**автоматично конвертує GLB у USDZ** для Quick Look (окремий файл не потрібен).

## Структура

```
testAr/
├── ar/
│   ├── index.html        # сторінка: селектор килимів + <model-viewer> + AR-кнопка
│   ├── carpet.glb        # прямокутний килим 1×2 м (JPEG, непрозорий) — за замовчуванням
│   ├── carpet1.glb       # килим 1.5×1.0 м з 1.png (PNG-alpha, BLEND)
│   └── carpet-round.glb  # ДЕМО: круглий килим з PNG-alpha (форма + м'який край)
├── make-glb.js           # генератор GLB: texture + розміри → .glb (без залежностей)
├── server.js             # статичний HTTP-сервер (0.0.0.0:4000), віддає .glb/.html/.jpg/.png
├── start-ngrok.js        # запуск ngrok-тунелю (HTTPS обов'язковий для AR)
├── test.jpg              # приклад прямокутної текстури
├── 1.png                 # приклад текстури-форми з прозорістю (3:2, RGBA)
├── sample-round.png      # приклад текстури-форми з прозорістю (круг, м'який край)
└── package.json
```

> Список килимів у селекторі задається масивом `CARPETS` у [ar/index.html](ar/index.html)
> (поля `name`, `size`, `glb`, `thumb`). Перший у масиві вантажиться за замовчуванням.
> Щоб додати килим: згенеруй GLB (див. нижче) і додай один рядок у `CARPETS`.

## Запуск і тест на телефоні

AR вимагає **HTTPS** → використовуємо ngrok.

```bash
# Термінал 1 — сервер
node server.js               # або: npm run dev

# Термінал 2 — тунель (ngrok.exe за шляхом D:\GIT\ngrok.exe, див. start-ngrok.js)
node start-ngrok.js          # або: npm run ngrok
# або одночасно обидва:
npm run dev:ngrok
```

ngrok видасть URL виду `https://xxxx.ngrok-free.app`. На телефоні відкрий:

```
https://xxxx.ngrok-free.app/ar/
```

> Перший раз ngrok-free показує сторінку-попередження → натисни **«Visit Site»**.

**На телефоні:** обери килим у селекторі зверху → «Переглянути в AR» → дозволь камеру →
наведи на підлогу → тапни, щоб поставити килим → рухай/обертай пальцем.

> На десктопі AR-кнопки не буде — лише 3D-перегляд мишкою. Це нормально, AR тільки на телефоні.

## Генерація GLB під конкретний килим

```bash
node make-glb.js <текстура> <ширина_м> <довжина_м> <вихід.glb> [alphaMode]

# приклади
node make-glb.js test.jpg 1 2 ar/carpet.glb                 # прямокутник, JPEG
node make-glb.js sample-round.png 1.4 1.4 ar/carpet-round.glb  # форма з PNG (auto→BLEND)
node make-glb.js rug-fringe.png 1.2 1.8 ar/rug.glb blend     # бахрома, м'який край
node make-glb.js logo-rug.png 1 1 ar/logo.glb mask           # чіткий виріз контуру
```

Генератор сам визначає тип (jpg/png) і наявність прозорості, вшиває байти текстури в GLB
без перекодування, робить матеріал двостороннім і піднімає килим на 5 мм над підлогою
(проти z-fighting).

### Форма та бахрома — через PNG з прозорістю (alpha)

Це головний механізм для **круглих/овальних/фігурних** килимів і для **бахроми (торочки)**:
форму й волосинки задає сам PNG, геометрія завжди лишається прямокутною площиною, а видно
тільки непрозорі пікселі.

**Як готувати текстуру (вимоги до дизайнера / контент-менеджера):**

1. Формат **PNG з alpha-каналом** (RGBA). JPEG прозорості не має — буде повний прямокутник.
2. Полотно PNG = **габаритний прямокутник килима, включно з бахромою**. Фон навколо килима — **повністю прозорий** (alpha = 0).
3. `<ширина_м>` і `<довжина_м>` у команді = реальний розмір цього **габаритного прямокутника** (з бахромою), а не лише тканої частини. Інакше масштаб в AR буде неправильний.
4. Співвідношення сторін PNG має збігатися зі співвідношенням `ширина:довжина`, інакше зображення розтягнеться (напр. круглий килим 1.4×1.4 м → квадратний PNG).
5. **Бахрома:** малюється по краях напівпрозорими «волосинками» з плавним згасанням alpha до 0 на кінчиках. Використовуй режим `blend`.
6. Орієнтація: верх PNG = далекий край килима від глядача (вісь −Z).

**Режими прозорості (`alphaMode`):**

| Режим | Коли | Поведінка |
|-------|------|-----------|
| `auto` (за замовч.) | універсально | PNG з alpha → `blend`; JPEG / PNG без alpha → `opaque` |
| `opaque` | звичайне прямокутне фото | alpha ігнорується, видно повний прямокутник |
| `mask` | чіткий контур (логотип-килим, різкий виріз) | піксель або повністю видимий, або повністю прозорий (поріг 0.5); без артефактів сортування |
| `blend` | **бахрома, м'які краї, плавна торочка** | справжня напівпрозорість; найкраще для волосинок |

> **Порада щодо AR:** `blend` дає найкращу торочку, але в рідкісних випадках на Android Scene Viewer
> напівпрозорі краї можуть мерехтіти (сортування прозорості). Якщо помітиш артефакти на конкретній
> моделі — перегенеруй її в `mask` (край буде різкішим, зате стабільним).

**Чого цей підхід НЕ дає:** справжнього об'ємного 3D-ворсу/бахроми з власною тінню — торочка
«намальована» на площині й під дуже гострим кутом виглядає пласкою. Для об'єму є наступні кроки
(не реалізовано): карта нормалей (`normalTexture`) для ілюзії ворсу під світлом, або окрема 3D-модель
для преміум-позицій. Для каталогу PNG+alpha — оптимальний баланс якість/масштабованість.

**Демо:** `ar/carpet-round.glb` (круглий, з `sample-round.png`) і `ar/carpet1.glb`
(з `1.png`) — приклади форми/прозорості. Усі три килими вже додані в селектор на сторінці,
перемикаються мініатюрами зверху.

## Вимоги пристроїв

| Платформа | Браузер | Умова |
|-----------|---------|-------|
| Android   | Chrome / Edge | встановлено **Google Play Services for AR (ARCore)** |
| iOS       | Safari  | iOS 12+ (ARKit, пристрій з підтримкою) |

HTTPS обов'язковий на всіх платформах.

## Інтеграція в Django-магазин (наступний крок)

Цей каталог — окремий прототип. Для впровадження в основний сайт:

1. На моделі `Product` (додаток `catalog`) додати поля:
   - `ar_texture = ImageField(upload_to="ar/tex/")` — **PNG з alpha** для фігурних килимів/бахроми
     (або звичайне фото для прямокутних), `width_m` / `length_m` — реальні розміри з бахромою.
   - `model_glb = FileField(upload_to="ar/", blank=True)` — згенерований результат.
2. При збереженні товару автоматично генерувати GLB з `ar_texture` + розмірів
   (логіку з `make-glb.js` нескладно перенести на Python — це прямий запис бінарного glTF,
   зовнішніх залежностей не треба). **Не забути перенести й визначення alpha:** читати тип
   зображення та `alphaMode` так само, як у `detectImage()`, інакше прозорість/форма зникнуть.
   Pillow (вже в `requirements.txt`) дає тип і канал альфа: `Image.open(f).mode == "RGBA"`.
3. У `templates/product.html` вставити (тільки якщо `product.model_glb` існує):
   ```html
   <model-viewer src="{{ product.model_glb.url }}" alt="{{ product.title }}"
     ar ar-modes="webxr scene-viewer quick-look" ar-placement="floor"
     ar-scale="fixed" camera-controls touch-action="pan-y" shadow-intensity="1">
     <button slot="ar-button">Переглянути в AR</button>
   </model-viewer>
   <script type="module"
     src="https://unpkg.com/@google/model-viewer@4.0.0/dist/model-viewer.min.js"></script>
   ```
4. Робочий сайт уже на HTTPS (Let's Encrypt) — додаткове налаштування для AR не потрібне.
